## Design Context

### Users
Developers and backend engineers testing HTTP APIs in the browser. They're focused, in flow, and context-switching between code and the tool constantly. The job to be done: send a request, understand the response, iterate fast. They want zero friction — no login, no bloat, no ceremony. Every second the UI gets in their way is a second lost.

### Brand Personality
**Clean. Modern. Approachable.**

Not a lab instrument, not a toy. Think: a well-made developer tool that respects your time and doesn't demand your attention. Confident without being cold. Technical without being intimidating.

### Reference & Anti-Reference
**Reference: [HTTPie for Web](https://httpie.io/app)**
What to steal from it: refined typography, generous-but-intentional whitespace, elegant method-color system, clean empty states, no visual noise, sidebar that breathes, smooth transitions that feel earned.

**Anti-reference: Postman**
What to avoid: toolbar bloat, too many surfaces competing for attention, icon-heavy navigation, aggressive upsell UI, modal-everything, panels that feel like they were bolted on over time. No feature creep in the UI.

### Aesthetic Direction
- **Tone:** Calm confidence. The interface should feel like a well-organized desk, not a cockpit.
- **Theme:** Both light and dark, equally polished. Light mode should feel crisp and airy (not washed out). Dark mode should feel deep and focused (not cave-dark).
- **Color:** Method-driven accent system is a strong differentiator — lean into it. Muted, desaturated method colors in the base state; accent brightens on interaction/focus.
- **Typography:** Inter for body/UI text. Geist Mono for all code, values, URLs, status codes, metrics. Let mono text signal "this is data" without needing extra decoration.
- **Density:** Medium — not as compressed as VS Code, not as spacious as a marketing site. Enough breathing room to scan quickly, tight enough to see context without scrolling.
- **Motion:** Purposeful only. Entrances on panel mounts, subtle transitions on state changes. No spinning loaders where a skeleton would do. Never animate for decoration.

### Design Principles
1. **Clarity over cleverness** — Every element earns its place. If it doesn't help the user understand or act, it doesn't exist.
2. **Data should feel precise** — Monospace for values, aligned columns, consistent number formatting. Numbers that look approximate undermine trust.
3. **Method color is personality** — The dynamic accent system is what makes this tool feel alive. Honor it consistently across every surface: tabs, badges, borders, focus rings, active states.
4. **Both themes are first-class** — Design for light and dark simultaneously. A component that only looks good in one mode isn't done.
5. **Density with rhythm** — Tight spacing within logical groups; generous separation between distinct sections. Space is a signal, not decoration.
